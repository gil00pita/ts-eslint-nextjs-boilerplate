# Patterns

Patterns describe reusable product flow shapes. Use them when a component alone is not enough to define the experience.

## Core Patterns

| Pattern                | Use when                                                          |
| ---------------------- | ----------------------------------------------------------------- |
| Empty state            | A user has no data, no results, or no permission to view content. |
| Error recovery         | A task failed and the user needs a next step.                     |
| Progressive disclosure | Details are useful but not always needed.                         |
| Review and confirm     | An action has meaningful consequences.                            |
| Filtering and search   | Users need to narrow a large data set.                            |
| Bulk action            | Users need to apply one action to multiple selected items.        |

## Pattern Anatomy

Each pattern should document:

- When to use it.
- When to avoid it.
- Required components.
- Required content.
- Keyboard and focus behavior.
- Empty, loading, success, and error states.

## Product Fit

Use the pattern as a starting point. Adjust density, language, and confirmation steps based on user risk, task frequency, and the cost of making a mistake.
