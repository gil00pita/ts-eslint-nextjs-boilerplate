# Accessibility

Accessibility is a baseline requirement in Phenix UI. Every product team is responsible for validating the final experience in context.

## Checklist

<table>
  <thead>
    <tr>
      <th>Area</th>
      <th>Check</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <td>Keyboard</td>
      <td>All interactive elements can be reached and operated without a pointer.</td>
    </tr>
    <tr>
      <td>Focus</td>
      <td>Focus order follows the visual and task order.</td>
    </tr>
    <tr>
      <td>Contrast</td>
      <td>Text, icons, borders, and focus indicators meet contrast requirements.</td>
    </tr>
    <tr>
      <td>Names</td>
      <td>Buttons, links, fields, and regions have clear accessible names.</td>
    </tr>
    <tr>
      <td>Errors</td>
      <td>Errors identify the field, explain the problem, and suggest recovery.</td>
    </tr>
    <tr>
      <td>Motion</td>
      <td>Non-essential animation respects reduced-motion preferences.</td>
    </tr>
  </tbody>
</table>

## Component Expectations

- Icon-only buttons must include an accessible name.
- Form inputs should use `Field.Root` so labels, help text, and errors are connected.
- Dialogs, menus, popovers, and selects should use their full compound structure.
- Status cannot be communicated by color alone.

## Review Moments

Run accessibility checks when:

- A new workflow is released.
- A component is added or changed.
- A route introduces custom keyboard behavior.
- Tokens or themes are updated.
